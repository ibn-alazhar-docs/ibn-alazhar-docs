# ADR-022: استبدال Google Vision API بمحرك OCR محلي (Surya)

## الحالة (Status)

Accepted

## السياق (Context)

ADR-007 السابق قرر استخدام Google Drive API كحل OCR لمرحلة MVP، مع التخطيط لـ Tesseract.js في V2. هذا القرار كان عملياً في وقت كتابته، لكنه يتعارض مع مبدأين أساسيين تبلورا لاحقاً في المشروع:

1. **Free-first (ADR-013):** الاعتماد على Google API يعني تبعية لخدمة خارجية بتكاليف متغيرة وحدود استخدام. هذا يناقض التوجه نحو استضافة ذاتية بتكلفة ~$6/شهر.
2. **Docker-first (ADR-019):** Google API يتطلب اتصال إنترنت و credentials خارجية، مما يعني أن بيئة التطوير لا يمكن أن تكون متسقة 100% مع الإنتاج.
3. **الخصوصية:** كتب التراث الإسلامي والمخطوطات لا ينبغي رفعها لخوادم Google مؤقتاً، حتى مع "حذف فوري". بعض المؤسسات التعليمية ترفض هذا صراحة.

بالإضافة إلى ذلك، نضجت محركات OCR مفتوحة المصدر للعربية بشكل كبير. **Surya** تحديداً يقدم:

- تحليل تخطيط (Layout Analysis) متقدم — يفهم الأعمدة، الهوامش، والحواشي
- دعم أصيل للعربية و RTL
- يعمل على CPU ولا يحتاج GPU
- جودة منافسة لـ Google Vision API في النصوص العربية المطبوعة

## القرار (Decision)

قررنا اعتماد **Surya** كمحرك OCR محلي وحيد في `workers/converter/`، يعمل داخل حاوية Docker على CPU، وإلغاء أي تكامل مع Google Drive/Cloud Vision API.

### تفاصيل القرار

1. **Surya** هو المحرك الأساسي والوحيد لمرحلة Phase 2 — لا Google API، لا Tesseract.js، لا Nougat.
2. المعالجة تكون بالكامل داخل حاوية العامل (worker container) عبر BullMQ queue.
3. لا رفع لأي ملف لخوادم خارجية في أي مرحلة من مراحل التحويل.
4. Arabic-Nougat يبقى خياراً للبحث المستقبلي (Phase 3+) فقط إذا توفر GPU في البنية التحتية.
5. Tesseract.js يُستبعد تماماً — أداؤه للعربية لا يرقى لمستوى الجودة المطلوب (≥95% دقة).

### هيكل خط الأنابيب الجديد

```
[رفع PDF → MinIO] → [BullMQ: conversion-queue] → [Worker: converter]
                                                       │
                                                       ├─ 1. pdf2image (PDF ← صفحات)
                                                       ├─ 2. Surya (تحليل تخطيط + OCR)
                                                       ├─ 3. Arabic-Reshaper (معالجة RTL)
                                                       ├─ 4. مصحح أساسي (قواعد)
                                                       └─ 5. تخزين extractedText في PostgreSQL
                                                       │
                                                       ▼
                                               [SSE: إشعار اكتمال]
```

## البدائل المعتبرة (Options Considered)

### 1. Surya محلياً (المختار)

- **المميزات:** جودة عربية ممتازة (≥95% للنصوص المطبوعة)، تحليل تخطيط متقدم، يعمل على CPU، لا تبعية خارجية، خصوصية كاملة، مفتوح المصدر (MIT)، مجتمع نشط، تثبيت عبر pip في حاوية Docker
- **العيوب:** أداء CPU أبطأ من GPU (لكن مقبول لـ 1-20 صفحة/جلسة)، يحتاج ذاكرة ≥2GB للنماذج، دقة أقل في المخطوطات اليدوية جداً (<70%)، لا يدوم طويلاً للمستندات الطويلة جداً (>200 صفحة)

### 2. Google Drive API (ADR-007 الأصلي)

- **المميزات:** جودة ممتازة، لا حاجة لموارد محلية
- **العيوب:** مخاوف خصوصية، تبعية خارجية، تكلفة متغيرة، حدود استخدام، لا يعمل offline
- **سبب الرفض:** يتعارض مع Free-first و Docker-first والخصوصية

### 3. Tesseract.js (مخطط سابقاً لـ V2)

- **المميزات:** مفتوح المصدر، يعمل محلياً في Node.js
- **العيوب:** دقة عربية متوسطة (~80-85%)، لا تحليل تخطيط، بطيء (WebAssembly)، يحتاج tuning مكثف للعربية
- **سبب الرفض:** دقة أقل من عتبة المشروع (95%)، لا يدعم تحليل التخطيط المعقد

### 4. Arabic-Nougat

- **المميزات:** جودة عالية، نموذج حديث
- **العيوب:** يحتاج GPU (VRAM ≥3GB)، غير متوفر على VPS $6/شهر، overkill لـ 90% من حالات الاستخدام
- **سبب الرفض:** غير قابل للتشغيل على البنية التحتية الحالية — مؤجل لـ Phase 3+

### 5. PaddleOCR

- **المميزات:** جودة جيدة، يدعم RTL
- **العيوب:** يحتاج Python runtime و GPU لأداء مثالي
- **سبب الرفض:** Surya يتفوق عليه في تحليل التخطيط ويعمل بكفاءة أعلى على CPU

### 6. EasyOCR

- **المميزات:** سهل التثبيت، يدعم العربية
- **العيوب:** دقة أقل من Surya، بطيء، لا تحليل تخطيط
- **سبب الرفض:** Surya يتفوق في كل المقاييس

## العواقب (Consequences)

### إيجابية

- **خصوصية كاملة:** لا ملف يغادر الخادم — يتوافق مع متطلبات المؤسسات التعليمية
- **تكلفة صفرية للتشغيل:** لا رسوم API، لا حدود استخدام، كل شيء داخل الحاوية
- **Docker-first متسق:** نفس البيئة في التطوير والإنتاج، اشتغل بدون إنترنت
- **تحكم كامل:** نتحكم في جودة OCR، التحديثات، والتحسينات دون انتظار طرف ثالث
- **توسع أفقي:** يمكن تشغيل عدة نسخ من العامل بالتوازي لزيادة الإنتاجية
- **يتوافق مع ADR-004:** Conversion (استخراج نص) منفصل عن Export (توليد صيغ)

### سلبية

- **أداء CPU:** تحويل مستند 10 صفحات قد يستغرق 30-60 ثانية (مقابل 5-10 ثوانٍ مع Google API)
- **حجم النماذج:** نماذج Surya تحتاج ~1-2GB في صورة Docker، مما يزيد حجم الحاوية
- **المخطوطات اليدوية:** الدقة تنخفض بشكل ملحوظ (<70%) — يحتاج حل منفصل لهذه الحالات (مستقبلاً)
- **صيانة إضافية:** تحديث نماذج OCR يتطلب بناء صورة Docker جديدة وإعادة نشر العامل

### مخاطر وتخفيفات

| المخاطرة                                | التأثير            | التخفيف                                                                         |
| --------------------------------------- | ------------------ | ------------------------------------------------------------------------------- |
| أداء CPU بطيء لمستخدمي الـ 20 تحويل/يوم | تجربة مستخدم بطيئة | إشعارات SSE فورية، تخزين النتائج، معالجة خلفية غير متزامنة                      |
| جودة OCR للمخطوطات اليدوية              | نتائج غير دقيقة    | إظهار نسبة الثقة للمستخدم، السماح بالتحرير اليدوي، وضع علامة على الثقة المنخفضة |
| نماذج Surya القديمة مع مرور الوقت       | تدهور الجودة       | CI/CD pipelining: إعادة بناء صورة العامل شهرياً مع آخر إصدارات Surya            |
| استهلاك ذاكرة 2GB للعامل الواحد         | ضغط على VPS        | تحديد maxMemory في الحاوية، تشغيل عامل واحد فقط على VPS CX22                    |

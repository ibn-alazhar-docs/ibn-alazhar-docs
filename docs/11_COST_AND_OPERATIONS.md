## 1. استراتيجية الأولوية للمجانية (Free-First)

### 1.1 المبادئ الأساسية

استراتيجية مستند تقوم على مبدأ جوهري: **الوصول المجاني أولاً (free-first) للمستخدم التعليمي**. لا يُطلب من المستخدم بطاقة ائتمان للوظائف الأساسية في التطوير المحلي، ولا يُجبر على خطة مدفوعة للميزات الجوهرية. الإنتاج العام قد يتطلب تكاليف VPS ونطاق. هذا يعكس القناعة بأن الأدوات التعليمية يجب أن تكون متاحة للجميع قدر الإمكان، خاصة في المنطقة العربية حيث الميزانيات محدودة.

المبادئ الخمسة للاستراتيجية:

| المبدأ | الوصف | التطبيق |
|--------|-------|---------|
| **لا بطاقة ائتمان للتطوير المحلي** | لا توجد خدمة في المكدس الأساسي تتطلب بطاقة ائتمان للتطوير والاختبار المحلي | MinIO بدل S3، PostgreSQL مُستضاف ذاتيًا بدل Supabase |
| **الحد الأدنى من التكاليف الثابتة** | لا مصاريف خفية أو اشتراكات سنوية مفاجئة | VPS واحد يُشغّل كل شيء، لا خدمة managed |
| **التحكم الكامل في البيانات** | البيانات على خوادمنا، لا على خوادم طرف ثالث | MinIO + PostgreSQL على نفس VPS |
| **الشفافية في التكاليف** | كل دولار مُبرر ومُوثّق | جدول تكاليف مفصّل في هذه الوثيقة |
| **الترقية عند الحاجة فقط** | لا ندفع إلا حين يقتضي النمو ذلك | VPS أكبر أو CDN عند تجاوز 500 مستخدم نشط |

### 1.2 معايير اتخاذ القرار: استضافة ذاتية أم خدمة مدفوعة

لكل خدمة جديدة نقيّم وفق المعايير التالية قبل اتخاذ قرار الاستضافة الذاتية مقابل الدفع لمزود:

| المعيار | استضافة ذاتية | خدمة مدفوعة |
|---------|-------------|-------------|
| **التكلفة الشهرية** | ضمن تكلفة VPS الحالي | اشتراك إضافي ثابت |
| **الحاجة لبطاقة ائتمان** | لا | غالبًا نعم |
| **جهد الصيانة** | متوسط-عالي | منخفض |
| **التحكم في البيانات** | كامل | محدود |
| **خطر التوقف** | يعتمد علينا | SLA من المزود |
| **القابلية للتوسع** | يدوية | تلقائية |
| **التوافق مع MIT License** | كامل | قد يتعارض |

**القاعدة**: إذا كانت الخدمة تتطلب بطاقة ائتمان ولا يوجد بديل مجاني مقبول للتطوير المحلي → نبحث عن بديل مفتوح المصدر نستضيفه ذاتيًا. إذا كان البديل الذاتي يتطلب جهد صيانة يفوق قدرة الفريق → نقبل الخدمة المدفوعة كاستثناء مع توثيق المبرر.

### 1.3 ما نستضيفه ذاتيًا مقابل ما ندفع مقابلها

| المكوّن | القرار | السبب |
|---------|--------|-------|
| PostgreSQL | **استضافة ذاتية** | بديل Supabase/PlanetScale يتطلب اشتراكًا مدفوعًا أو بطاقة ائتمان |
| MinIO | **استضافة ذاتية** | بديل S3 يتطلب بطاقة ائتمان وتكلفة حسب الاستخدام غير متوقعة |
| Redis | **استضافة ذاتية** | بديل Upstash/Redis Cloud المجاني محدود جدًا (256MB) |
| Next.js App | **استضافة ذاتية** | بديل Vercel Pro مدفوع؛ الحصة المجانية لا تكفي لـ SSE + Workers |
| Caddy | **استضافة ذاتية** | SSL مجاني عبر Let's Encrypt، لا حاجة لـ Cloudflare Paid |
| Google Drive OCR | **خدمة خارجية مجانية** | الحصة المجانية تكفي لـ MVP، لا بديل مجاني مكافئ للعربية |
| GitHub | **مجاني للمشاريع المفتوحة** | CI/CD مجاني، لا حاجة لخطة مدفوعة |
| UptimeRobot | **مجاني** | 50 monitor مجانًا، يكفي تمامًا |

## 2. استراتيجية الاستضافة الذاتية (Self-Hosting)

### 2.1 الفوائد

| الفائدة | التفصيل |
|---------|---------|
| **تكلفة ثابتة ومتوقعة** | VPS بـ 5-20$/شهر يُشغّل كل شيء، لا مفاجآت |
| **سيادة كاملة على البيانات** | ملفات المستخدمين على MinIO الخاص بنا، لا على AWS |
| **لا قيود المزود** | لا rate limiting مفاجئ، لا تعليق حساب، لا شروط استخدام غامضة |
| **تعلم الفريق** | تشغيل البنية التحتية يعزز مهارات DevOps لدى الفريق |
| **توافق مع رخصة MIT** | المستخدم يمكنه استضافة المنصة على خادمه الخاص بسهولة |
| **خصوصية أعلى** | بيانات المستخدمين لا تمر عبر مزودين طرف ثالث (عدا Google OCR المؤقت) |

### 2.2 المخاطر

| الخطر | الاحتمالية | التأثير | التخفيف |
|-------|-----------|---------|---------|
| **انقطاع الخادم** | متوسط | عالي — انقطاع كامل | UptimeRobot للإنذار، Caddy auto-restart، systemd services |
| **فقدان البيانات** | منخفض | كارثي | نسخ احتياطي يومي لـ PostgreSQL + MinIO، احتفاظ لـ 7 أيام |
| **امتلاء القرص** | متوسط | عالي — فشل الرفع والتحويل | تنبيه عند 80%، تنظيف تلقائي للملفات المؤقتة، حصص مستخدمين |
| **هجوم أمني** | منخفض | عالي | Caddy rate limiting، CSRF protection، CSP headers، تحديثات منتظمة |
| **فشل الأجهزة** | منخفض | كارثي | مزود VPS موثوق (Hetzner/Contabo)، نسخ احتياطي خارجي |
| **توقف المزود** | منخفض | عالي | اختيار مزود بـ SLA ≥ 99.9%، خطة نقل جاهزة |

### 3.2 تفصيل تكاليف Google Drive API

Google Drive API يُستخدم لـ OCR عبر آلية "upload image → export as text". الحصة المجانية تعتمد على Google Cloud Project:

| المورد | الحصة المجانية | ما يعادله في Ibn Al-Azhar Docs |
|--------|---------------|------------------|
| **Drive API Queries** | 1,000,000,000/يوم | غير محدود عمليًا |
| **Drive API Upload** | 10,000,000/يوم | ~10,000 ملف/يوم |
| **Drive API Download** | 10,000,000/يوم | ~10,000 ملف/يوم |
| **OCR Processing** | لا حد صريح منفصل | محدود بـ upload quota |
| **Project Quota** | 300 requests/minute/user | 5 طلبات/ثانية لكل مستخدم |

**التكلفة الفعلية**: مجانًا ضمن الحصة المجانية لـ Google Cloud Project. لا يوجد معدّل مدفوع لـ Google Drive API نفسه. القيود هي:
- الحصة لكل مشروع (queries per 100 seconds)، ويمكن زيادتها عبر طلب من Google Console
- **يتطلب إعداد Google Cloud Project** مع مراجعة الحصص
- **سياسات Google قد تتغير** — يجب مراجعة الحصص دورياً
- الحصة المجانية تكفي لـ MVP ولكن يجب مراقبة الاستخدام مع نمو القاعدة

## 6. سياسة الاستخدام العادل

### 6.1 ما يُشكّل إساءة استخدام

المنصة مصمّمة للاستخدام التعليمي الشخصي. يُعتبر استخدمًا غير عادل:

| نوع الإساءة | الوصف | مثال |
|-------------|-------|------|
| **الاستخدام التجاري المكثّف** | استخدام المنصة كخدمة OCR تجارية دون إذن | شركة مسح ضوئي تُحوّل 500 ملف/يوم |
| **التخزين غير التعليمي** | تخزين ملفات لا علاقة لها بالتعليم | رفع أفلام أو برمجيات |
| **المشاركة غير المشروعة** | مشاركة محتوى محمي بحقوق الطبع | رفع كتاب محفوظ الحقوق ومشاركته |

```typescript
// lib/quota-check.ts
async function checkStorageQuota(userId: string, additionalBytes: number): Promise<void> {
  const user = await prisma.user.findUnique({
    where: { id: userId },
    select: { role: true, storageUsed: true },
  });

  const quota = STORAGE_QUOTA[user.role]; // 1GB / 2GB / 10GB
  const projectedUsage = user.storageUsed + additionalBytes;

  if (projectedUsage > quota) {
    throw new AppError('STORAGE_QUOTA_EXCEEDED', 413, {
      used: formatBytes(user.storageUsed),
      quota: formatBytes(quota),
      needed: formatBytes(additionalBytes),
    });
  }
}
```

```typescript
// lib/abuse-detector.ts
interface AbuseSignal {
  userId: string;
  signalType: 'RAPID_UPLOAD' | 'MASS_CONVERSION' | 'MULTI_ACCOUNT' | 'UNUSUAL_PATTERN';
  severity: 'low' | 'medium' | 'high';
  details: Record<string, unknown>;
}

async function detectAbuse(userId: string): Promise<AbuseSignal[]> {
  const signals: AbuseSignal[] = [];
  const now = new Date();

  // 1. رفع سريع غير عادي
  const recentUploads = await prisma.file.count({
    where: { userId, createdAt: { gte: subMinutes(now, 10) } },
  });
  if (recentUploads > 50) {
    signals.push({ userId, signalType: 'RAPID_UPLOAD', severity: 'high', details: { count: recentUploads } });
  }

  // 2. تحويلات مكثفة
  const recentConversions = await prisma.conversion.count({
    where: { userId, createdAt: { gte: subHours(now, 1) } },
  });
  if (recentConversions > 30) {
    signals.push({ userId, signalType: 'MASS_CONVERSION', severity: 'medium', details: { count: recentConversions } });
  }

  // 3. حسابات متعددة من نفس IP
  const userIp = await prisma.loginLog.findFirst({ where: { userId } });
  if (userIp) {
    const accountsFromSameIp = await prisma.loginLog.count({
      where: { ipAddress: userIp.ipAddress, createdAt: { gte: subDays(now, 1) } },
      distinct: ['userId'],
    });
    if (accountsFromSameIp > 3) {
      signals.push({ userId, signalType: 'MULTI_ACCOUNT', severity: 'high', details: { accountsCount: accountsFromSameIp } });
    }
  }

  return signals;
}
```

```
الخطوة 2: فحص الاتصالات النشطة
  $ docker compose exec postgres psql -U ibn_al_azhar_docs -c "
    SELECT count(*), state FROM pg_stat_activity GROUP BY state;
  "
  # هل هناك connections عالقة؟
  # هل max_positions قريب من الامتلاء؟

الخطوة 3: إنهاء الاتصالات العالقة
  $ docker compose exec postgres psql -U ibn_al_azhar_docs -c "
    SELECT pg_terminate_backend(pid)
    FROM pg_stat_activity
    WHERE state = 'idle' AND query_start < now() - interval '30 minutes';
  "

الخطوة 4: فحص الاستعلامات البطيئة
  $ docker compose exec postgres psql -U ibn_al_azhar_docs -c "
    SELECT query, mean_exec_time, calls
    FROM pg_stat_statements
    ORDER BY mean_exec_time DESC
    LIMIT 10;
  "

الخطوة 5: إعادة تشغيل PostgreSQL (كاخير خيار)
  $ docker compose restart postgres
  # انتظر 30 ثانية
  $ docker compose exec postgres pg_isready

الخطوة 6: استعادة من النسخ الاحتياطي (في حالة تلف البيانات)
  $ scp backup@backup-server:ibn_al_azhar_docs_$(date +%Y%m%d).sql.gz /tmp/
  $ gunzip /tmp/ibn_al_azhar_docs_*.sql.gz
  $ docker compose exec -T postgres psql -U ibn_al_azhar_docs < /tmp/ibn_al_azhar_docs_*.sql
```

**الحد الزمني**: 10 دقائق للتشخيص، 30 دقيقة لاستعادة النسخة الاحتياطية

## 12. عملية الدعم

### 12.1 قنوات الدعم

| القناة | الغرض | وقت الاستجابة المتوقع | الأولوية |
|--------|-------|---------------------|---------|
| **GitHub Issues** | تقارير الأخطاء، طلبات الميزات | 48 ساعة | متوسطة |
| **البريد الإلكتروني** | مشاكل الحسابات، أسئلة خاصة | 24 ساعة | عالية |
| **Discord/Telegram** | دعم مجتمعي سريع | غير مضمون | منخفضة |
| **Documentation** | أسئلة شائعة، أدلة استخدام | ذاتي | — |

### 12.2 مستويات التصعيد

```
المستوى 1: ذاتي
  └─ المستخدم يبحث في التوثيق والأسئلة الشائعة
  └─ حل معظم المشاكل الشائعة

المستوى 2: مجتمعي
  └─ طرح السؤال على Discord/Telegram
  └─ مستخدمون آخرون + فريق المتطوعين

المستوى 3: فريق الدعم
  └─ فتح GitHub Issue أو إرسال بريد
  └─ استجابة خلال 24-48 ساعة

المستوى 4: مهندس
  └─ مشاكل تقنية معقدة تُحوّل للفريق التقني
  └─ استجابة خلال 4 ساعات عمل

المستوى 5: إدارة
  └─ مشاكل أمنية أو قانونية
  └─ استجابة فورية
```

### 12.3 أنواع طلبات الدعم

| النوع | المثال | المستوى | وقت الحل المستهدف |
|-------|--------|---------|------------------|
| **استفسار** | كيف أُغيّر لغة الواجهة؟ | 1-2 | ذاتي |
| **مشكلة استخدام** | التحويل فشل، ماذا أفعل؟ | 2-3 | 24 ساعة |
| **خطأ تقني** | الصفحة لا تتحمّل، خطأ 500 | 3-4 | 48 ساعة |
| **مشكلة حساب** | لا أستطيع تسجيل الدخول | 3-4 | 24 ساعة |
| **طلب ميزة** | أريد دعم صيغة XLSX | 2-3 | غير محدد (أولوية المنتج) |
| **حادث أمني** | أعتقد أن حسابي مُخترق | 4-5 | فوري |

### 12.4 اعتبارات SLA

باعتبار المنصة مجانية ومفتوحة المصدر، لا نقدم SLA مُلزم قانونيًا. لكننا نلتزم داخليًا بالأهداف التالية:

| المقياس | الهدف | الملاحظة |
|---------|-------|---------|
| **Uptime** | ≥ 99% (شهرًا) | ~7 ساعات توقف شهريًا مقبولة |
| **وقت الاستجابة للدعم** | ≤ 48 ساعة | لأول رد على GitHub Issue/Email |
| **وقت حل المشاكل التقنية** | ≤ 5 أيام عمل | للمشاكل غير الحرجة |
| **وقت احتواء الحوادث الأمنية** | ≤ 30 دقيقة | للاحتواء الأولي |
| **وقت استعادة الخدمة** | ≤ 4 ساعات | من لحظة اكتشاف الانقطاع |

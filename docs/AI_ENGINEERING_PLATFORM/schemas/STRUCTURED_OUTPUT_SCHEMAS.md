# Structured Output Schemas

> **Purpose:** Define JSON schemas for all AI-generated outputs in the AI Engineering Platform.
> **Scope:** Plans, tasks, reviews, verification results, repair reports, phase gates, PR summaries, runtime metrics.

---

## 1. Schema Registry

All schemas are registered in a central registry and validated at runtime:

```typescript
interface SchemaRegistry {
  schemas: Map<string, JSONSchema>;
  get(name: string): JSONSchema;
  validate(name: string, data: unknown): ValidationResult;
}
```

## 2. Plan Schemas

### 2.1 ImplementationPlan

Generated by the architect session during the planning stage.

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "ImplementationPlan",
  "type": "object",
  "required": ["specId", "branchName", "tasks", "estimatedDurationMinutes", "risks"],
  "properties": {
    "specId": {
      "type": "string",
      "description": "ID of the spec being implemented"
    },
    "branchName": {
      "type": "string",
      "pattern": "^[a-z0-9][a-z0-9-]*[a-z0-9]$",
      "description": "Git branch name for this implementation"
    },
    "tasks": {
      "type": "array",
      "items": { "$ref": "#/definitions/Task" },
      "minItems": 1,
      "description": "Ordered list of implementation tasks"
    },
    "estimatedDurationMinutes": {
      "type": "integer",
      "minimum": 1,
      "maximum": 30,
      "description": "Estimated implementation time"
    },
    "risks": {
      "type": "array",
      "items": { "$ref": "#/definitions/Risk" },
      "description": "Identified risks and mitigations"
    },
    "dependencies": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Specs or PRs this implementation depends on"
    },
    "filesToCreate": {
      "type": "array",
      "items": { "type": "string" },
      "description": "New files to be created"
    },
    "filesToModify": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Existing files to be modified"
    }
  },
  "definitions": {
    "Task": {
      "type": "object",
      "required": ["id", "description", "files", "acceptanceCriteria"],
      "properties": {
        "id": { "type": "string" },
        "description": { "type": "string" },
        "files": {
          "type": "array",
          "items": { "type": "string" }
        },
        "acceptanceCriteria": {
          "type": "array",
          "items": { "type": "string" }
        }
      }
    },
    "Risk": {
      "type": "object",
      "required": ["description", "severity", "mitigation"],
      "properties": {
        "description": { "type": "string" },
        "severity": { "enum": ["low", "medium", "high", "critical"] },

      }
    }
  }
}
```

## 15. Schema Evolution

### 11.1 Versioning

- Each schema has a `version` field in the event.
- Breaking changes increment the major version.
- Non-breaking changes increment the minor version.
- Old versions are supported during transition periods.

### 11.2 Backward Compatibility Rules

| Change | Compatibility | Action |
|---|---|---|
| Add optional field | Backward compatible | Bump minor version |
| Add enum value | Backward compatible | Bump minor version |
| Remove field | Breaking | Bump major version, migration required |
| Change field type | Breaking | Bump major version, migration required |
| Add required field | Breaking | Bump major version, migration required |

### 11.3 Migration Strategy

1. New schema version is registered alongside old version.
2. Both versions are accepted during transition.
3. Old version is deprecated after 30 days.
4. Old version is removed after 90 days.

## 3. دليل المطور

### الإعداد المحلي (Setup)

```bash
# 1. استنساخ المستودع
git clone https://github.com/org/ibn-al-azhar-docs.git
cd ibn-al-azhar-docs

# 2. تثبيت المتطلبات
npm install

# 3. إعداد متغيرات البيئة
cp .env.example .env
# تعبئة القيم: DATABASE_URL, MINIO_*, GOOGLE_DRIVE_*, NEXTAUTH_*, REDIS_URL

# 4. تشغيل Docker (PostgreSQL + Redis + MinIO)
docker compose up -d

# 5. تهيئة قاعدة البيانات (DB: ibn_al_azhar_docs)
npx prisma migrate dev
npx prisma generate
npx prisma db seed

# 6. تشغيل التطبيق
npm run dev          # Next.js على المنفذ 3000
npm run dev:worker   # BullMQ worker
```

## 4. توثيق API

### طريقة التوليد

| الخيار | الوصف | التوصية |
|---|---|---|
| OpenAPI/Swagger UI | توليد تلقائي من JSDoc أو zod schemas | ✅ مُوصى به لـ MVP |
| TypeDoc | توثيق TypeScript تلقائي | للمكتبات الداخلية |
| Markdown يدوي | مرونة كاملة لكن جهد عالي | للـ endpoints المعقدة فقط |

### مكان الاستضافة

- **أثناء التطوير**: `/api/docs` endpoint مع Swagger UI
- **للمساهمين**: في مجلد `docs/api/` داخل المستودع
- **للمستخدمين**: لا حاجة (MVP لا يعرض API عام)

### هيكل توثيق كل Endpoint

```markdown
### POST /api/files/upload

**الوصف**: رفع ملف أو أكثر إلى المنصة

**المصادقة**: مطلوبة (JWT session)

**الحدود**: 
- الحد الأقصى: 100MB
- الأنواع: application/pdf, image/png, image/jpeg, image/webp

**الطلب**:
- Content-Type: multipart/form-data
- الحقول: file (مطلوب), folderId (اختياري)

**الاستجابة الناجحة** (200):
\```json
{
  "id": "uuid",
  "name": "document.pdf",
  "size": 1048576,
  "status": "uploaded"
}
\```

**أخطاء محتملة**:
- 401: غير مصادق
- 413: الملف أكبر من 100MB
- 415: نوع ملف غير مدعوم
```

## 6. دليل استكشاف الأخطاء

### مشاكل الرفع

| المشكلة | السبب المحتمل | الحل |
|---|---|---|
| "الملف كبير جداً" | تجاوز 100MB | ضغط الملف أو تقسيمه |
| "نوع ملف غير مدعوم" | ملف غير PDF/صورة | تحويل لـ PDF أولاً |
| الرفع يتوقف عند % | مشكلة شبكة أو timeout | زيادة timeout في Caddy |
| "فشل الرفع" بدون تفاصيل | MinIO غير متصل | `docker compose restart minio` + فحص السجلات |

### مشاكل التحويل (OCR)

| المشكلة | السبب المحتمل | الحل |
|---|---|---|
| التحويل لا يعمل | Google Drive API quota استنفدت | الانتظار حتى اليوم التالي أو طلب زيادة الحصة |
| نتائج OCR فارغة | ملف تالف أو صورة غير واضحة | تجربة ملف بجودة أعلى |
| OCR بطيء جداً | ضغط على خوادم Google | الانتظار أو إعادة المحاولة |
| النص العربي مشوّه | مشكلة ترميز | فحص encoding في ملف PDF الأصلي |
| "فشل المعالجة" | BullMQ worker معطّل | `docker compose restart worker` |

### مشاكل التصدير

| المشكلة | السبب المحتمل | الحل |
|---|---|---|
| DOCX لا يفتح | مشكلة في مكتبة docx | تجربة صيغة TXT كـ بديل |
| نص عربي مشوّه في TXT | مشكلة ترميز | فتح الملف بترميز UTF-8 |
| ملف التصدير غير موجود | انتهت صلاحية الملف المؤقت | إعادة التصدير |

### مشاكل التسجيل والدخول

| المشكلة | السبب المحتمل | الحل |
|---|---|---|
| "البريد مسجل" | حساب موجود | تسجيل الدخول بدلاً من ذلك |
| جلسة منتهية | انتهاء صلاحية JWT (24 ساعة) | تسجيل الدخول مجدداً |
| "حساب مقفل" | 5 محاولات فاشلة | الانتظار 15 دقيقة |

### مشاكل PWA

| المشكلة | السبب المحتمل | الحل |
|---|---|---|
| لا يظهر Install Prompt | لا HTTPS أو manifest ناقص | التأكد من HTTPS + manifest |
| التطبيق لا يعمل offline | Service Worker لم يُسجّل | فحص Application → SW في DevTools |
| أيقونة التطبيق غير واضحة | أيقونة صغيرة جداً | استخدام أيقونة 512x512 |
| لا يعمل على Safari | Apple meta tags ناقصة | إضافة `apple-mobile-web-app-capable` |

### مشاكل الأداء

| المشكلة | السبب المحتمل | الحل |
|---|---|---|
| الصفحة بطيئة | حمولة كبيرة على الخادم | فحص `docker stats` + إضافة موارد |

## 7. الأسئلة الشائعة (FAQ)

### عام

**س1: ما هو ابن الأزهر دوكس (Ibn Al-Azhar Docs)؟**
ج: ابن الأزهر دوكس هي منصة ويب عربية لإدارة المستندات التعليمية وتحويلها. تتيح لك رفع ملفات PDF والصور وتحويلها إلى نص قابل للبحث والتعديل عبر تقنية OCR (استخراج النص)، ثم تصديرها بصيغ متعددة (توليد الصيغ).

**س2: هل ابن الأزهر دوكس مجاني؟**
ج: نعم، ابن الأزهر دوكس مجاني للاستخدام الأساسي. يمكنك رفع وتحويل وتصدير المستندات بدون رسوم. المنصة مفتوحة المصدر برخصة MIT ويمكن استضافتها ذاتياً.

**س3: ما المتصفحات المدعومة؟**
ج: ابن الأزهر دوكس يعمل على Chrome (آخر إصدارين)، Firefox (آخر إصدارين)، Safari (آخر إصدارين)، وEdge (آخر إصدارين). يُنصح باستخدام Chrome للحصول على أفضل تجربة.

**س4: هل يمكنني استخدام ابن الأزهر دوكس على الجوال؟**
ج: نعم! ابن الأزهر دوكس مصمم كـ PWA (تطبيق ويب تقدمي) ويعمل على جميع الأجهزة. يمكنك تثبيته على شاشتك الرئيسية كتطبيق مستقل — راجع دليل تثبيت PWA.

### الرفع والملفات

**س5: ما أنواع الملفات التي يمكنني رفعها؟**
ج: يمكنك رفع ملفات PDF وصور (PNG, JPEG, WebP). الحد الأقصى لحجم الملف هو 100MB.

**س6: كم عدد الملفات التي يمكنني رفعها؟**
ج: لا يوجد حد لعدد الملفات في MVP. حدود معقولة فقط لمنع إساءة الاستخدام (rate limiting).

**س7: هل يمكنني رفع ملفات تحتوي نصاً عربياً؟**
ج: بالتأكيد! ابن الأزهر دوكس مصمم كمنصة عربية أولاً ويدعم التعرف على النص العربي عبر OCR مع أولوية عالية للغة العربية.

### التحويل والتصدير

**س8: ما الفرق بين التحويل والتصدير؟**
ج: **التحويل** هو عملية استخراج النص من ملف PDF أو صورة عبر تقنية OCR. **التصدير** هو عملية توليد ملف بصيغة معينة (TXT, DOCX, JSON) من النص المُستخرج. هاتان عمليتان منفصلتان — يمكنك التحويل ثم التصدير بصيغة مختلفة في أي وقت.

**س9: ما هو OCR؟**
ج: OCR (التعرف البصري على الحروف) هو تقنية تحول النص في الصور وملفات PDF إلى نص رقمي قابل للبحث والتعديل. راجع قسم شرح OCR للمزيد.

**س10: كم يستغرق التحويل؟**
ج: يعتمد على حجم الملف وعدد الصفحات. عادةً: صفحة واحدة = 5-10 ثوانٍ، 10 صفحات = 20-30 ثانية، 50 صفحة = 1-2 دقيقة.

**س11: لماذا كانت نتيجة OCR غير دقيقة؟**
ج: دقة OCR تعتمد على جودة الملف الأصلي. لتحسين النتائج: استخدم ملفات بجودة عالية، تأكد من وضوح النص، وتجنب الصور المشوّشة أو المائلة.

**س12: ماذا يحدث لملفاتّ أثناء OCR؟**
ج: تُرسل ملفاتك مؤقتاً إلى Google Drive API للمعالجة، ثم تُحذف من خوادم Google فوراً. لا يتم الاحتفاظ بنسخة على خوادم Google. راجع قسم الخصوصية.

### الخصوصية والأمان

**س13: أين تُخزن ملفاتي؟**
ج: تُخزن ملفاتك على خادم MinIO خاص بالمنصة (bucket: ibn-al-azhar-docs-files). إذا كنت تستضيف ذاتياً، فالملفات على خادمك الخاص. البيانات مشفّرة في النقل عبر HTTPS.

**س14: هل يمكن لأي شخص رؤية ملفاتي؟**
ج: لا. ملفاتك خاصة ولا يمكن لأي مستخدم آخر الوصول إليها إلا إذا شاركتها عبر رابط عام. روابط المشاركة عشوائية (64 hex chars) ولا يمكن تخمينها.

**س15: كيف أحذف حسابي وبياناتي؟**
ج: من صفحة الإعدادات، اختر "حذف الحساب". سيتم حذف جميع ملفاتك وبياناتك نهائياً خلال 30 يوماً. لا يمكن التراجع عن هذا الإجراء.

### PWA والاستخدام

**س16: كيف أثبّت التطبيق على جهازي؟**
ج: راجع دليل تثبيت PWA في القسم 10 أدناه للحصول على خطوات مفصلة لكل متصفح.

**س17: هل يعمل التطبيق بدون إنترنت؟**
ج: جزئياً. يمكنك تصفح الملفات المحفوظة مسبقاً. لكن الرفع والتحويل والتسجيل يتطلبون اتصالاً بالإنترنت.

### استضافة ذاتية (Self-Hosting)

**س18: كيف أستضيف ابن الأزهر دوكس على خادمي الخاص؟**
ج: ابن الأزهر دوكس مفتوح المصدر ويمكن نشره عبر Docker Compose مع Caddy كـ reverse proxy. راجع دليل النشر في `docs/19_DEPLOYMENT_GUIDE.md` للحصول على خطوات مفصلة.

**س19: ما متطلبات الخادم؟**
ج: VPS بـ 2 cores + 4GB RAM + 40GB SSD كحد أدنى. يُنصح بـ 4 cores + 8GB RAM + 100GB SSD.

## 8. شرح الخصوصية للمستخدمين

### الرسالة الأساسية

> ملفاتك بأمان. نحن نحترم خصوصيتك ونبذل كل جهد لحماية بياناتك.

### كيف نشرح الخصوصية للمستخدم

#### ملفاتك تُخزن بأمان
- تُخزن ملفاتك على خوادم خاصة بنا (MinIO) وليس على خدمات تخزين عامة
- جميع الاتصالات مشفّرة عبر HTTPS — لا يمكن اعتراض ملفاتك أثناء النقل
- لا يمكن لأي مستخدم آخر الوصول لملفاتك إلا إذا شاركتها بنفسك

#### ماذا يحدث عند التحويل (OCR)
- عندما تطلب تحويل ملف، يُرسل مؤقتاً إلى Google Drive API للمعالجة
- Google تعالج الملف وتُرجع النص فوراً
- **لا تحتفظ Google بنسخة من ملفك** بعد المعالجة
- يمكننا استخدام بديل محلي في المستقبل

#### ماذا يحدث عند التصدير
- التصدير عملية محلية على خوادمنا — لا تُرسل بياناتك لأي طرف ثالث
- ملفات التصدير تُخزن مؤقتاً ثم تُحذف بعد 24 ساعة

#### حذف البيانات
- يمكنك حذف أي ملف في أي وقت
- الملفات المحذوفة تبقى في سلة المحذوفات 30 يوماً ثم تُحذف نهائياً
- يمكنك حذف حسابك بالكامل مع جميع بياناتك

#### من يصل لبياناتك
- فريق التطوير: لا يطلع على ملفاتك الشخصية
- Google Drive API: يرى الملف مؤقتاً أثناء المعالجة فقط
- لا نشارك بياناتك مع أي طرف ثالث آخر

#### مشاركة الملفات
- الروابط العامة عشوائية (64 hex chars) — لا يمكن تخمينها
- يمكنك إلغاء المشاركة في أي وقت
- لا توجد إحصائيات تتبع للروابط في MVP

### نص صفحة الخصوصية (مختصر)

```
نحن في ابن الأزهر دوكس نأخذ خصوصيتك على محمل الجد:
✅ ملفاتك مشفّرة في النقل (HTTPS)
✅ لا نشارك بياناتك مع أطراف ثالثة (ما عدا Google Drive API للـ OCR)
✅ يمكنك حذف حسابك وبياناتك بالكامل
✅ ملفاتك المحذوفة تُحذف نهائياً بعد 30 يوماً
✅ روابط المشاركة عشوائية وآمنة (64 hex chars)
⚠️ ملفاتك تُرسل مؤقتاً لـ Google للمعالجة عبر OCR
```

## 9. شرح تقنية OCR

### ما هو OCR؟

**OCR** (Optical Character Recognition — التعرف البصري على الحروف) هو تقنية تحوّل النص الموجود في الصور أو ملفات PDF الممسوحة ضوئياً إلى نص رقمي قابل للبحث والنسخ والتعديل.

### كيف يعمل OCR في ابن الأزهر دوكس

```
┌──────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────────┐
│ رفع الملف │───▶│ BullMQ Queue │───▶│ Google Drive │───▶│ حفظ النتيجة  │
│ (PDF/صورة)│    │ (ترتيب+معالجة)│    │    API       │    │ (PostgreSQL) │
└──────────┘    └──────────────┘    └─────────────┘    └──────────────┘
                                          │
                                    ┌─────▼──────┐
                                    │ حذف مؤقت   │
                                    │ من Google   │
                                    └────────────┘
```

#### الخطوات بالتفصيل

| الخطوة | ماذا يحدث | المدة |
|---|---|---|
| 1. رفع المستخدم | يُرفع الملف إلى MinIO ويُسجّل في PostgreSQL | ثوانٍ |
| 2. إضافة للطابور | تُضاف مهمة OCR إلى BullMQ queue في Redis | فوري |
| 3. معالجة Worker | يأخذ Worker الملف من MinIO ويرسله لـ Google Drive API | 5-60 ثانية |
| 4. استلام النتيجة | يستلم Worker النص من Google ويحفظه في PostgreSQL | ثوانٍ |
| 5. إشعار المستخدم | يظهر النص في واجهة المستخدم | فوري |

### التحويل مقابل التصدير

| العملية | ماذا تفعل | متى |
|---|---|---|
| **التحويل (Conversion)** | استخراج النص من ملف PDF/صورة عبر OCR | مرة واحدة لكل ملف |
| **التصدير (Export)** | توليد ملف بصيغة محددة (TXT/DOCX/JSON) من النص المُستخرج | أي وقت بعد التحويل |

### ما يحدث لملفك على Google

| المرحلة | ماذا يحدث | المدة |
|---|---|---|
| رفع مؤقت | يُرفع الملف إلى Google Drive الخاص بالخدمة | أثناء المعالجة فقط |
| معالجة OCR | Google يستخرج النص من الملف | 5-60 ثانية |
| حذف فوري | يُحذف الملف من Google Drive بعد استلام النتيجة | فوري |
| **لا احتفاظ** | Google لا تحتفظ بنسخة من الملف | — |

### دقة OCR للغة العربية

| نوع المستند | الدقة المتوقعة | ملاحظات |
|---|---|---|
| نص عربي واضح (طباعة حاسوب) | 95-99% | أفضل النتائج |
| نص عربي ممسوح ضوئياً (جيد) | 85-95% | يعتمد على الجودة |
| نص عربي بخط يدوي | 50-70% | غير مُوصى به |
| نص مختلط (عربي + إنجليزي) | 80-90% | قد يكون هناك أخطاء |
| نص مع تشكيل | 75-85% | التشكيل قد يضيع أحياناً |

### تأثير الخصوصية

- **ما يراه Google**: محتوى الملف أثناء المعالجة فقط
- **ما يحتفظ به Google**: لا شيء — الملف يُحذف فوراً بعد المعالجة
- **ما يراه فريق ابن الأزهر دوكس**: بيانات وصفية فقط (اسم الملف، حجمه، حالة المعالجة) — لا المحتوى
- **البديل المستقبلي**: محرك OCR محلي (Tesseract) يُشغّل على الخادم بدون إرسال بيانات خارجية

## 10. دليل تثبيت PWA

### ما هو PWA؟

PWA (Progressive Web App — تطبيق الويب التقدمي) هو تطبيق ويب يعمل كتطبيق أصلي على جهازك. يظهر أيقونة على شاشتك الرئيسية، يعمل بنافذة مستقلة بدون شريط عنوان المتصفح، ويمكنه العمل بدون إنترنت جزئياً.

> **ملاحظة**: نحن نقيّم استخدام @serwist/next كأداة PWA — لم يُحسم القرار النهائي بعد.

### التثبيت على Chrome (Android + سطح المكتب)

#### على Android

1. افتح موقع ابن الأزهر دوكس في Chrome
2. ستظهر بانر "إضافة إلى الشاشة الرئيسية" تلقائياً
3. اضغط **"تثبيت"** أو **"إضافة إلى الشاشة الرئيسية"**
4. تأكيد التثبيت
5. ستجد أيقونة ابن الأزهر دوكس على شاشتك الرئيسية

> إذا لم تظهر البانر: اضغط على ⋮ (القائمة) → "تثبيت التطبيق" أو "إضافة إلى الشاشة الرئيسية"

#### على سطح المكتب (Chrome)

1. افتح موقع ابن الأزهر دوكس في Chrome
2. ستظهر أيقونة تثبيت في شريط العنوان (على اليمين)
3. اضغط عليها → **"تثبيت"**
4. أو: اضغط ⋮ (القائمة) → "تثبيت ابن الأزهر دوكس..."
5. سيفتح التطبيق في نافذة مستقلة

### التثبيت على Firefox (Android + سطح المكتب)

#### على Android

1. افتح موقع ابن الأزهر دوكس في Firefox
2. اضغط على ⋮ (القائمة)
3. اختر **"تثبيت"** أو **"إضافة إلى الشاشة الرئيسية"**
4. تأكيد التثبيت

#### على سطح المكتب

1. افتح موقع ابن الأزهر دوكس في Firefox
2. اضغط على أيقونة التثبيت في شريط العنوان
3. أو: اضغط ☰ (القائمة) → "تثبيت هذا الموقع كتطبيق"
4. تأكيد التثبيت

> ⚠️ دعم Firefox للـ PWA على سطح المكتب محدود. يُنصح باستخدام Chrome للحصول على أفضل تجربة.

### التثبيت على Safari (iOS + macOS)

#### على iOS (iPhone/iPad)

1. افتح موقع ابن الأزهر دوكس في Safari
2. اضغط على أيقونة المشاركة (في الأسفل)
3. مرر للأسفل واختر **"إضافة إلى الشاشة الرئيسية"**
4. تأكيد الاسم والضغط على **"إضافة"**
5. ستجد أيقونة ابن الأزهر دوكس على شاشتك الرئيسية

> ⚠️ على iOS، التطبيق يعمل داخل Safari WebView. بعض ميزات PWA قد تكون محدودة مقارنة بـ Android.

#### على macOS (Safari 17+)

1. افتح موقع ابن الأزهر دوكس في Safari
2. من شريط القائمة: ملف → **"إضافة إلى Dock"**
3. أو: اضغط على أيقونة المشاركة → "إضافة إلى Dock"
4. تأكيد التثبيت

### التثبيت على Edge

1. افتح موقع ابن الأزهر دوكس في Edge
2. ستظهر أيقونة تثبيت في شريط العنوان
3. أو: اضغط ⋯ (القائمة) → **"تثبيت هذا الموقع كتطبيق"**
4. تأكيد التثبيت

### إلغاء تثبيت PWA

| المتصفح | كيفية الإلغاء |
|---|---|
| Chrome (Android) | إعدادات الجهاز → التطبيقات → ابن الأزهر دوكس → إلغاء التثبيت |
| Chrome (سطح المكتب) | داخل التطبيق: ⋮ → "إلغاء تثبيت ابن الأزهر دوكس" |
| Safari (iOS) | اضغط مطولاً على الأيقونة → "إزالة التطبيق" |
| Edge | داخل التطبيق: إعدادات → "إلغاء تثبيت" |
| Firefox | إعدادات الجهاز → التطبيقات → ابن الأزهر دوكس → إلغاء التثبيت |

## 11. دليل المساهمة

### سير العمل (Workflow)

```
1. Fork المستودع (ibn-al-azhar-docs)
2. إنشاء فرع: git checkout -b feat/short-description
3. التطوير + الاختبارات
4. التأكد من اجتياز lint + tests: npm run lint && npx vitest run
5. Commit: git commit -m "feat: add file preview component"
6. Push: git push origin feat/short-description
7. إنشاء Pull Request مع وصف مفصل
8. مراجعة الكود (Code Review)
9. التعديل حسب الملاحظات
10. Merge بعد الموافقة
```

### معايير الكود

| المعيار | التفاصيل |
|---|---|
| TypeScript strict | لا `any` بدون تعليق `// eslint-disable-next-line @typescript-eslint/no-explicit-any -- reason` |
| مكونات React | functional components + hooks فقط |
| أنماط | Tailwind CSS فقط — لا CSS مخصص إلا للـ RTL |
| حالة التطبيق | Zustand للـ global state، React state للـ local |
| جلب البيانات | Next.js Server Components + API routes |
| أخطاء | Error boundaries + toast notifications |
| إمكانية الوصول | semantic HTML + ARIA labels + keyboard navigation |
| اختبارات | Vitest للـ unit/integration — NOT Jest |

### عملية Pull Request

| العنصر | المطلوب |
|---|---|
| العنوان | Conventional Commit format: `feat:`, `fix:`, `docs:`, etc. |
| الوصف | ماذا تغير + لماذا + كيف تم اختباره |
| اختبارات | Unit tests للتغييرات الجديدة (Vitest) |
| توثيق | تحديث التوثيق ذي الصلة |
| حجم PR | أقل من 400 سطر — قسّم التغييرات الكبيرة |

### معايير المراجعة (Review Criteria)

| المعيار | الأسئلة |
|---|---|
| الصحة | هل الكود يعمل كما هو متوقع؟ |
| الأمان | هل هناك ثغرات محتملة؟ (XSS, SQLi, IDOR) |
| الأداء | هل هناك استعلامات بطيئة أو عمليات غير ضرورية؟ |
| القراءة | هل الكود مفهوم بدون شرح إضافي؟ |
| الاختبارات | هل تغطي اختبارات Vitest الحالات الجديدة؟ |
| التوافق | هل يعمل مع RTL + اللغة العربية؟ |
| التوثيق | هل تم تحديث التوثيق ذي الصلة؟ |

### الإبلاغ عن الأخطاء (Bug Reports)

استخدم template التالي:

```markdown
## وصف الخطأ
وصف موجز للمشكلة

## خطوات الإعادة
1. اذهب إلى '...'
2. اضغط على '...'
3. شاهد الخطأ

## السلوك المتوقع
ماذا كان يجب أن يحدث

## السلوك الفعلي
ماذا حدث فعلاً

## لقطات شاشة
(إذا وجدت)

## بيئة التشغيل
- المتصفح: [مثلاً Chrome 120]
- الجهاز: [مثلاً Pixel 7 / Desktop]
- نظام التشغيل: [مثلاً Android 14 / Windows 11]
```

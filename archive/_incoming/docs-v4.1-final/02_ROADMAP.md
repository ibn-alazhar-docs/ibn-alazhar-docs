# Ibn Al-Azhar Docs — ابن الأزهر دوكس — خارطة الطريق

> **الحالة:** مراجعة نهائية
> **ارتباط:** ملخّص المشروع (00_PROJECT_BRIEF.md) · وثيقة المتطلبات (01_PRD.md)

## نظرة عامة

خارطة طريق ابن الأزهر دوكس (Ibn Al-Azhar Docs) مصمّمة لتقديم منتج قابل للاستخدام بأسرع وقت ممكن، مع بناء أساس تقني متين يدعم التوسع في الإصدارات اللاحقة. الخارطة تبدأ بمرحلة نموذج أولي (Prototype) مدتها أسبوع واحد، تليها مرحلة MVP مقسّمة إلى 6 sprints مدة كل منها أسبوع، ثم سلسلة من الإصدارات التوسعية (V1، V2، V3) تمتد حتى 9 أشهر بعد الإطلاق، وأخيرًا رؤية مؤسسية/مستقبلية.

الفلسفة الأساسية هي: **شحن قيمة حقيقية في كل مرحلة**. لا ننتظر الكمال قبل الإطلاق، لكننا لا نشحن منتجًا غير مكتمل الوظائف الأساسية. كل sprint ينتج ميزات يمكن اختبارها واستخدامها فورًا، وكل إصدار لاحق يبني على الأساس السابق دون إعادة بناء.

## المرحلة 0: النموذج الأولي (Prototype) — الأسبوع 0-1

### الهدف

إثبات جدوى المفهوم التقني الأساسي: هل يمكننا تحويل ملف PDF ممسوح إلى نص عربي عبر خط أنابيب الويب (pdfjs-dist → Google Drive OCR → تصدير)؟ هل تعمل مكوّنات shadcn/ui بشكل صحيح في وضع RTL؟ هل يمكننا تشغيل BullMQ وMinIO محليًا؟ هذه المرحلة تجيب على الأسئلة الحرجة قبل الالتزام ببناء MVP كامل.

### الميزات

- إعداد بيئة التطوير المحلية (Docker Compose لـ PostgreSQL + Redis + MinIO) `[ new-proposal ]`
- نموذج أولي لخط أنابيب التحويل: ملف PDF واحد → استخراج صفحة واحدة عبر pdfjs-dist → OCR عبر Google Drive API → تصدير TXT `[ needs-verification ]`
- صفحة ويب بسيطة مع Next.js 16 لاختبار الرفع والتحويل
- اختبار دعم RTL في عدة مكوّنات shadcn/ui (Button، Input، Dialog، Table) `[ design-decision ]`
- إعداد BullMQ مع مهمة واحدة وإثبات أنها تعمل من البداية للنهاية `[ new-proposal ]`
- اختبار اتصال MinIO ورفع/تحميل ملف
- إعداد Caddy كوكيل عكسي محلي `[ decided-ADR-001 ]`

### ما هو خارج النطاق

- المصادقة وتسجيل الدخول
- واجهة مستخدم متكاملة أو تصميم نهائي
- قاعدة البيانات وPrisma ORM
- التدويل (i18n)
- أي ميزة إنتاجية (PWA، إشعارات، لوحة إدارة)
- التعامل مع الملفات الكبيرة أو متعددة الصفحات

### التبعيات

- الحصول على مفاتيح Google Drive API وتمكين OCR endpoint
- تثبيت Docker وDocker Compose على أجهزة التطوير
- تأكيد إصدارات المكتبات (Next.js 16، pdfjs-dist، BullMQ) متوافقة
- إعداد Caddy للبيئة المحلية

### المخاطر

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| Google Drive API لا يوفّر جودة OCR كافية للعربية | منخفضة | أثبت tahweel ذلك مسبقًا كمرجع بحثي، لكن نعيد التحقق |
| pdfjs-dist بطيء جدًا في استخراج الصفحات | متوسطة | قياس الأداء على ملف 10 صفحات؛ إذا فشل، نقيّم بدائل خادمية |
| مشاكل في تشغيل MinIO محليًا | منخفضة | استخدام Docker image الرسمي مع إعدادات موثقة |

### معايير النجاح

1. تحويل ملف PDF من صفحة واحدة إلى نص عربي قابل للقراءة بنجاح
2. خط أنابيب BullMQ يعمل من البداية للنهاية مع إشعارات حالة
3. ثلاثة مكوّنات shadcn-ui على الأقل تعمل بشكل صحيح في RTL
4. رفع وتحميل ملف من/إلى MinIO بنجاح
5. قياس أداء أولي: استخراج صفحة PDF + OCR < 30 ثانية

## المرحلة 1: MVP — الأسابيع 1-6

### Phase 1: الأساس (الأسبوع 1)

**الهدف:** إعداد البنية التحتية للمشروع وقاعدة البيانات والمصادقة والتدويل ونظام التصميم والتخطيط العام. هذا Phase يؤسّس كل ما ستبني عليه المراحل اللاحقة، لذا يجب أن يكون متينًا وموثّقًا جيدًا.

**الميزات:**

- إنشاء مستودع Next.js 16 مع هيكل مجلدات واضح (app/, components/, lib/, types/) `[ design-decision ]`
- إعداد Prisma ORM مع PostgreSQL: نماذج المستخدم (User) والدور (Role) والملف (File) والمجلد (Folder) والمهمة (Job) `[ new-proposal ]` — قاعدة البيانات: `ibn_al_azhar_docs`
- تنفيذ المصادقة عبر NextAuth.js v5: تسجيل، دخول، خروج، إدارة الجلسات (JWT strategy، maxAge 24h)، أدوار (طالب/معلم/مشرف) — لا تدفق access/refresh token منفصل `[ new-proposal ]`
- إعداد next-intl مع ملفات ترجمة عربية وإنجليزية وتوجيه RTL/LTR تلقائي `[ design-decision ]`
- تثبيت وتهيئة shadcn/ui + Tailwind CSS v4 مع دعم RTL `[ design-decision ]`
- إنشاء التخطيط الرئيسي (Main Layout): شريط جانبي + شريط علوي + منطقة محتوى مع تنقل سلس
- صفحة هبوط أساسية تشرح المنتج مع زرّي تسجيل ودخول
- إعداد ESLint + Prettier + Husky pre-commit hooks
- إعداد Vitest + React Testing Library كأدوات اختبار `[ decided ]`

**ما هو خارج النطاق:**

- رفع الملفات وتحويلها (Phase 2-3)
- لوحة الإدارة (Phase 5)
- Service Worker وPWA (Phase 5)
- أي ميزة متقدمة

**التبعيات:**

- إكمال مرحلة النموذج الأولي بنجاح
- قاعدة بيانات PostgreSQL عاملة (محليًا أو عن بعد)
- قرار حول حدود التخزين لكل مستخدم (Q3 في PRD)

**المخاطر:**

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| مشاكل تكامل NextAuth.js v5 مع Prisma | متوسطة | استخدام القوالب الرسمية من NextAuth |
| بعض مكوّنات shadcn/ui لا تدعم RTL بشكل كامل | عالية | اختبار فوري وتعديل CSS يدويًا `[ design-decision ]` |
| تعقيد إعداد next-intl مع App Router | متوسطة | اتباع التوثيق الرسمي والقوالب المرجعية |

**معايير النجاح:**

1. مستخدم يمكنه إنشاء حساب وتسجيل الدخول والخروج (JWT strategy، maxAge 24h)
2. الواجهة تتبدل بين العربية (RTL) والإنجليزية (LTR) بشكل صحيح
3. التخطيط الرئيسي متجاوب ويعمل على الهاتف والحاسوب
4. نماذج Prisma مُطبّقة وقاعدة البيانات مُهاجرة (migrated) بنجاح

### Phase 2: إدارة الملفات (الأسبوع 2)

**الهدف:** بناء نظام إدارة الملفات الكامل: الرفع، المجلدات الهرمية، سلة المحذوفات، التحميل، والبحث الأساسي. هذا Phase يمنح المستخدم مساحة عمل حقيقية لتنظيم مستنداته قبل حتى أن يبدأ التحويل.

**الميزات:**

- رفع الملفات (PDF + JPG/PNG) مع سحب وإفلات، حد 100MB، تخزين في MinIO (bucket: `ibn-al-azhar-docs-files`) `[ new-proposal ]` `[ ADR-014 ]`
- رفع عدة ملفات دفعة واحدة مع عرض تقدّم الرفع لكل ملف
- إنشاء مجلدات هرمية: إنشاء، إعادة تسمية، نقل ملفات بينها، حذف `[ new-proposal ]`
- عرض الملفات والمجلدات: قائمة وشبكة (list/grid view)، ترتيب، تصفية
- سلة المحذوفات: حذف ناعم، استعادة، حذف نهائي `[ new-proposal ]`
- تحميل الملفات المرفوعة من MinIO
- بحث أساسي في أسماء الملفات والمجلدات `[ new-proposal ]`
- إشعارات Toast عبر Sonner لجميع العمليات (نجاح، خطأ، تحذير) `[ new-proposal ]`

**ما هو خارج النطاق:**

- التحويل وخط الأنابيب (Phase 3)
- المشاركة (Phase 4)
- إعدادات المستخدم المتقدمة

**التبعيات:**

- إكمال Phase 1 (المصادقة، قاعدة البيانات، التخطيط)
- MinIO يعمل ويمكن الرفع إليه والتحميل منه

**المخاطر:**

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| رفع ملفات كبيرة (100MB) يفشل أو ينتهي وقته | متوسطة | زيادة timeout في Next.js API routes؛ استخدام chunked upload |
| واجهة نظام الملفات معقدة في RTL | متوسطة | اختبار كل عرض (قائمة/شبكة) مع نص عربي طويل |
| MinIO لا يتسع للاختبار المحلي | منخفضة | إعداد حجم مناسب في Docker |

**معايير النجاح:**

1. رفع ملف PDF وملف صورة بنجاح مع رؤيتهما في قائمة الملفات
2. إنشاء مجلد ونقل ملف إليه
3. حذف ملف واستعادته من سلة المحذوفات
4. البحث عن ملف بالاسم والحصول على نتيجة صحيحة
5. رفع 5 ملفات دفعة واحدة بنجاح

### Phase 3: خط أنابيب التحويل (الأسبوع 3)

**الهدف:** بناء المحرك الخلفي لخط أنابيب التحويل بالكامل: BullMQ، pdfjs-dist، Google Drive OCR، وSSE لتحديث التقدّم. هذا Phase هو قلب المنصة التقني ويجب أن يعمل بشكل موثوق. التحويل (Conversion) يعني استخراج النص عبر OCR فقط. التصدير (Export) يبنى في خطوة لاحقة.

**الميزات:**

- إعداد BullMQ مع Redis: تعريف الطوابير، معالجات المهام، آلية إعادة المحاولة (retry)، التعامل مع الأخطاء (dead letter queue) `[ new-proposal ]`
- مرحلة Preparing: التحقق من صحة الملف، قراءة البيانات الوصفية، إنشاء سجل المهمة في قاعدة البيانات `[ repo-reference ]`
- مرحلة Splitting: استخراج صفحات PDF كصور عبر pdfjs-dist `[ new-proposal ]` `[ needs-verification ]`
- مرحلة OCR: إرسال الصور إلى Google Drive API واستقبال النص `[ doc-verified ]`
- مرحلة Writing: تجميع النصوص وحفظ نتيجة التحويل في قاعدة البيانات `[ repo-reference ]`
- مرحلة Done: تحديث حالة المهمة، إشعار المستخدم
- نقطة نهاية SSE (Server-Sent Events) لبث تحديثات حالة التحويل في الوقت الفعلي `[ new-proposal ]`
- إعدادات المستخدم: DPI، التزامن، فاصل الصفحات عبر Zustand `[ repo-reference ]`

**ما هو خارج النطاق:**

- واجهة مستخدم للتحويل (Phase 4)
- التصدير بصيغ متعددة (Phase 4)
- المشاركة (Phase 4)
- لوحة الإدارة (Phase 5)

**التبعيات:**

- إكمال Phase 2 (رفع الملفات وتخزينها)
- Google Drive API key صالح ويعمل (مؤكد من Prototype)
- Redis يعمل ويقبل اتصالات BullMQ

**المخاطر:**

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| pdfjs-dist بطيء جدًا في Splitting لملفات كبيرة | متوسطة | قياس الأداء؛ تقييم معالجة خادمية بـ Rust كخطة ب `[ needs-verification ]` |
| Google Drive API يرفض طلبات OCR متزامنة | متوسطة | تنفيذ rate limiting محلي؛ تجنب إرسال أكثر من 5 طلبات متوازية |
| SSE تفقد الاتصال في الشبكات الضعيفة | متوسطة | آلية إعادة الاتصال التلقائية في المتصفح + fallback لـ polling |

**معايير النجاح:**

1. تحويل ملف PDF من 10 صفحات إلى نص عربي في أقل من 60 ثانية
2. نتيجة التحويل (النص المستخرج) محفوظة في قاعدة البيانات
3. تحديثات SSE تصل للمتصفح في الوقت الفعلي مع كل مرحلة
4. إعادة محاولة مهمة فاشلة بنجاح
5. إعدادات المستخدم (DPI، فاصل الصفحات) تُحترم في التحويل

### Phase 4: واجهة التحويل + التصدير + المشاركة (الأسبوع 4)

**الهدف:** بناء واجهة المستخدم الكاملة للتحويل: بدء التحويل، متابعة التقدّم، تصدير النتائج بصيغ متعددة (خطوة منفصلة عن التحويل)، وتنفيذ نظام المشاركة عبر الروابط العامة. هذا Phase يربط المحرك الخلفي (Phase 3) بالواجهة الأمامية ويكمل رحلة المستخدم الأساسية.

**الميزات:**

- صفحة التحويل: زر "تحويل" بجانب كل ملف، عرض المراحل مع وصف عربي (جاري التحضير ← جاري التقسيم ← جاري التعرف ← جاري الكتابة ← مكتمل) `[ new-proposal ]`
- متابعة تقدّم التحويل في الوقت الفعلي عبر SSE مع شريط تقدّم مرئي ورسائل حالة `[ new-proposal ]`
- عرض نتيجة التحويل: نص مستخرج مع تنسيق
- تصدير النتائج (Export): خطوة منفصلة من نتيجة التحويل، أزرار تصدير بصيغ متعددة (TXT/DOCX/JSON) `[ repo-reference ]`
- إعادة محاولة التحويل الفاشل مع عرض رسالة الخطأ `[ new-proposal ]`
- إنشاء رابط مشاركة عام لملف محوّل (عرض + تحميل) — رمز crypto-random 32 bytes hex-encoded `[ new-proposal ]`
- صفحة المشاركة العامة: عرض النص وخيارات التصدير دون تسجيل دخول `[ new-proposal ]`
- إلغاء رابط المشاركة `[ new-proposal ]`
- قائمة الملفات المشتركة لكل مستخدم
- إشعارات Toast لجميع الأحداث (بدء التحويل، اكتمال، فشل، مشاركة)

**ما هو خارج النطاق:**

- المشاركة مع مستخدمين محددين (V2)
- معاينة PDF (V2)
- لوحة الإدارة (Phase 5)

**التبعيات:**

- إكمال Phase 3 (خط أنابيب التحويل يعمل)
- إكمال Phase 2 (نظام الملفات)

**المخاطر:**

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| واجهة تقدّم التحويل مربكة للمستخدم | متوسطة | اختبار usability مع 3 مستخدمين حقيقيين |
| روابط المشاركة العامة تُستغل لتحميل مفرط | متوسطة | تطبيق rate limiting على روابط المشاركة؛ إضافة خيار انتهاء الصلاحية |
| نص OCR طويل لا يعرض بشكل جيد في المتصفح | منخفضة | تصفح مقسّم (pagination) أو lazy loading للنصوص الطويلة |

**معايير النجاح:**

1. المستخدم يستطيع بدء تحويل ومتابعة التقدّم لحظيًا ورؤية النتيجة
2. تصدير نتيجة التحويل بصيغ TXT وDOCX وJSON بنجاح
3. إنشاء رابط مشاركة وفتحه من جهاز آخر دون تسجيل دخول
4. إلغاء رابط مشاركة والتأكد من توقف العمل به
5. إعادة محاولة تحويل فاشل بنجاح

### Phase 5: الإدارة + PWA (الأسبوع 5)

**الهدف:** بناء لوحة الإدارة الأساسية للمشرفين وتحويل المنصة إلى تطبيق ويب تقدّمي (PWA) قابل للتثبيت. هاتان الميزتان تكملان تجربة MVP: الإدارة تضمن استقرار المنصة، وPWA يوفّر تجربة تطبيق أصلي على الأجهزة المحمولة.

**الميزات:**

لوحة الإدارة:

- لوحة إحصائيات رئيسية: عدد المستخدمين، التحويلات اليومية/الأسبوعية، معدل النجاح/الفشل، استهلاك التخزين `[ new-proposal ]`
- إدارة المستخدمين: قائمة مع بحث وتصفية، تغيير الأدوار، تعطيل/تفعيل الحسابات `[ new-proposal ]`
- مراقبة الطوابير: عدد المهام في كل طابور (BullMQ)، المهام الفاشلة، معدل المعالجة
- تقييد الوصول: لوحة الإدارة متاحة فقط للمشرفين

PWA:

- Web App Manifest: اسم التطبيق، أيقونات متعددة الأحجام، لون السمة، اتجاه العرض، start_url `[ new-proposal ]`
- Service Worker: مقترح عبر @serwist/next (يحتاج spike للتأكيد النهائي)، تخزين App Shell مؤقتًا باستخدام استراتيجية Cache First `[ new-proposal ]` `[ needs-spike ]`
- Install Prompt مخصص: رسالة تثبيت تظهر بعد الزيارة الثالثة مع زر "تثبيت التطبيق" `[ new-proposal ]`
- العمل دون اتصال الأساسي: عرض App Shell ورسالة "أنت غير متصل" عند فقدان الاتصال
- أيقونات PWA بأحجام متعددة (192×192، 512×512) مصمّمة للهوية البصرية لابن الأزهر دوكس

**ما هو خارج النطاق:**

- Background Sync (V2)
- Web Push Notifications (V3)
- Offline data access متقدم
- إحصائيات تفصيلية وتقارير (V3)
- إدارة المنصة المتقدمة (حصص API، سجلات النظام)

**التبعيات:**

- إكمال Phase 1-4 (المنصة تعمل بشكل كامل)
- تصميم أيقونات PWA (يمكن استخدام أيقونات مؤقتة)
- spike لـ @serwist/next للتأكد من ملاءمته

**المخاطر:**

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| @serwist/next لا يتوافق مع Next.js 16 | متوسطة | إجراء spike مبكر؛ fallback لـ next-pwa أو service worker يدوي |
| Service Worker يُخزّن ملفات قديمة ولا يحدّثها | متوسطة | استخدام استراتيجية Stale-While-Revalidate للمحتوى الديناميكي |
| Install Prompt لا يظهر في بعض المتصفحات | متوسطة | كشف دعم beforeinstallprompt وعرض رسالة بديلة |
| لوحة الإدارة تستغرق وقتًا أكبر من المتوقع | متوسطة | تقييد النطاق على الإحصائيات الأساسية وإدارة المستخدمين فقط |

**معايير النجاح:**

1. المشرف يرى إحصائيات محدّثة في لوحة الإدارة
2. المشرف يغيّر دور مستخدم بنجاح
3. التطبيق يُثبَّت على هاتف Android عبر رسالة Install Prompt المخصصة
4. App Shell يُعرض دون اتصال بعد التثبيت
5. Lighthouse PWA audit يمر جميع الفحوصات الأساسية

### Phase 6: الاختبار + الإطلاق (الأسبوع 6)

**الميزات (اختبار وإعداد):**

- توثيق الإعداد والتشغيل (deployment guide)
- إعداد CI/CD pipeline أساسي (GitHub Actions)

**ما هو خارج النطاق:**

- تحسينات الأداء المتقدمة (V1)

**التبعيات:**

**المخاطر:**

## المرحلة 2: V1 — الشهران 2-3 بعد MVP

### الهدف

تثبيت المنتج ومعالجة المشاكل المكتشفة بعد الإطلاق الفعلي. V1 لا يضيف ميزات جديدة جوهرية بل يركز على جودة التجربة: إصلاح الأخطاء، تحسين الأداء، تبسيط واجهة المستخدم بناءً على ملاحظات حقيقية، وتحسين الاستقرار التشغيلي. هذه المرحلة ضرورية لتحويل MVP من نموذج أولي usable إلى منتج production-ready يحظى بثقة المستخدمين.

### الميزات

- **إصلاح الأخطاء (Bug Fixes):** معالجة جميع الأخطاء المبلّغ عنها من المستخدمين الأوائل، مع أولوية للأخطاء التي تمنع إتمام رحلة المستخدم الأساسية (رفع ← تحويل ← تصدير) `[ new-proposal ]`
- **تحسين الأداء:** تحسين وقت التحميل الأول (FCP < 1.5s)، تحسين استجابة واجهة نظام الملفات مع أعداد كبيرة من الملفات، تحسين استهلاك الذاكرة في خط أنابيب التحويل `[ new-proposal ]`
- **تحسين تجربة المستخدم:** تبسيط تدفقات العمل بناءً على ملاحظات المستخدمين، تحسين الرسائل العربية (وضوح ودقة)، تحسين تجربة الهاتف المحمول خاصة في الرفع والتحويل `[ new-proposal ]`
- **مراقبة محسّنة:** إضافة Application Performance Monitoring (APM)، تنبيهات استباقية عند اقتراب حصة Google API من النفاد، مراقبة مساحة MinIO مع تنبيهات `[ new-proposal ]`
- **اختبار الحمل:** اختبار المنصة مع 100 مستخدم متزامن وتحديد نقاط الاختناق `[ needs-verification ]`
- **توثيق المستخدم:** صفحة مساعدة ومقالات إرشادية أساسية بالعربية `[ new-proposal ]`

### ما هو خارج النطاق

- ميزات جديدة (مخصصة لـ V2)
- إعادة هيكلة معمارية كبيرة
- دعم لغات جديدة
- واجهة برمجة تطبيقات عامة

### التبعيات

- ملاحظات المستخدمين من MVP (حقيقية وليس افتراضية)
- أدوات مراقبة (مقترح: Sentry للأخطاء، Uptime Robot للتوفر)

### المخاطر

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| ردود فعل سلبية من المستخدمين الأوائل تبطئ التبني | متوسطة | استجابة سريعة للمشاكل؛ تواصل شفاف مع المجتمع |
| تحسينات الأداء تتطلب تغييرات معمارية | منخفضة | قياس قبل التحسين؛ تغيير تدريجي |

### معايير النجاح

1. جميع الأخطاء الحرجة والمتوسطة مُعالجة
2. Lighthouse Performance score > 95 في الإنتاج
3. متوسط وقت التحويل < 45 ثانية لـ 10 صفحات (تحسين من 60)
4. معدل الاحتفاظ (7-day retention) > 40%
5. 200 مستخدم مسجّل بحلول نهاية V1

## المرحلة 3: V2 — الشهران 3-4 بعد MVP

### الهدف

توسيع قدرات المنصة بشكل جوهري: إضافة OCR محلي كبديل عن Google Drive API، وتصدير بصيغ جديدة، وبحث نصي كامل، وتتبع إصدارات الملفات، ومعاينة PDF، ومشاركة مع مستخدمين محددين، وإشعارات بريد إلكتروني، ودعم أفضل للعمل دون اتصال. V2 يحوّل ابن الأزهر دوكس من أداة تحويل أساسية إلى منصة إدارة مستندات متكاملة تقلّل الاعتماد على الخدمات الخارجية وتزيد القيمة للمستخدم.

### الميزات

- **OCR محلي عبر Tesseract.js:** تشغيل OCR في المتصفح أو على الخادم كبديل عن Google Drive API. هذا يُعالج خطر استنفاد حصة API ويوفّر خصوصية أكبر للمستندات الحساسة. التحدي الرئيسي هو جودة OCR العربية مقارنةً بـ Google `[ needs-verification ]`
- **تصدير Markdown وHTML:** إضافة صيغتي Markdown وHTML كخيارات تصدير بجانب TXT وDOCX وJSON. مفيد للمعلمين الذين يريدون نشر المحتوى على مواقع تعليمية `[ new-proposal ]`
- **البحث بالنص الكامل (Full-Text Search):** البحث داخل محتوى الملفات المحوّلة وليس فقط في أسماء الملفات. يتطلب محرك بحث مناسب (مقترح: PostgreSQL Full-Text Search أو Meilisearch كخيار منفصل) `[ new-proposal ]`
- **تتبع الإصدارات (Version Tracking):** حفظ تاريخ تعديلات الملف مع إمكانية العودة لإصدار سابق. مفيد عندما يُعاد تحويل ملف بإعدادات مختلفة ويريد المستخدم مقارنة النتائج `[ new-proposal ]`
- **معاينة PDF:** عرض ملف PDF في المتصفح دون تحميله، مع إمكانية التنقل بين الصفحات. يتطلب مُعرّض PDF مدمج (pdfjs-dist viewer) `[ new-proposal ]`
- **المشاركة مع مستخدمين محددين:** مشاركة ملف أو مجلد مع مستخدمين مسجّلين محددين بدلًا من رابط عام فقط، مع أذونات (عرض فقط / عرض + تحميل) `[ new-proposal ]`
- **إشعارات البريد الإلكتروني:** إرسال بريد عند اكتمال التحويل، وعند مشاركة ملف مع المستخدم، وعند اقتراب حصة التخزين `[ new-proposal ]`
- **Background Sync:** مزامنة الملفات المرفوعة دون اتصال عند استعادة الاتصال عبر Service Worker Background Sync API `[ new-proposal ]`

### ما هو خارج النطاق

- نظام المؤسسات (V3)
- واجهة برمجة تطبيقات عامة (V3)
- تطبيق سطح مكتب (V3)
- تكامل LMS (V3)

### التبعيات

- مكتبة Tesseract.js مع نماذج اللغة العربية (تحتاج تحميل ~20MB)
- محرك بحث (PostgreSQL FTS أو Meilisearch)
- خدمة إرسال بريد (مقترح: Resend أو Nodemailer مع SMTP)

### المخاطر

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| Tesseract.js لا يوفر جودة OCR عربية مقبولة | عالية | اختبار مبكر؛ إبقاء Google Drive API كخيار أساسي وTesseract كبديل |
| محرك البحث يبطئ قاعدة البيانات | متوسطة | استخدام Meilisearch المنفصل بدلًا من PostgreSQL FTS إذا لزم |
| حجم ميزات V2 كبير ويستغرق أكثر من شهرين | متوسطة | تقسيم V2 إلى V2a وV2b إذا لزم؛ إعطاء أولوية لـ OCR المحلي والبحث |

### معايير النجاح

1. Tesseract OCR يعمل وينتج نصًا عربيًا مقبولًا (دقة > 80%) مع إمكانية التبديل بينه وبين Google
2. البحث بالنص الكامل يُرجع نتائج صحيحة خلال ثانيتين
3. مشاركة ملف مع مستخدم مسجّل يعمل مع الأذونات الصحيحة
4. 500 مستخدم مسجّل بحلول نهاية V2
5. معدل نجاح التحويل > 95%

## المرحلة 4: V3 — الأشهر 6-9 بعد MVP

### الهدف

تحويل ابن الأزهر دوكس من منصة فردية إلى منصة مؤسسية: نظام المؤسسات والأقسام، أرشفة تلقائية، تحليلات وتقارير، تكامل مع أنظمة إدارة التعلم (LMS)، واجهة برمجة تطبيقات عامة، ودعم صيغ إدخال جديدة، وتحسينات OCR بالذكاء الاصطناعي، وتصنيف تلقائي للمستندات، وأخيرًا تطبيق سطح مكتب عبر Tauri. V3 يوسع نطاق المنصة من الأفراد إلى المؤسسات ويفتح إمكانيات التكامل مع النظام التعليمي الأوسع.

### الميزات

- **نظام المؤسسات (Organization System):** إنشاء مساحات مؤسسية بأقسام وأعضاء وأذونات هرمية. كل مؤسسة لها مساحة اسم (namespace) خاصة وإدارة مستقلة `[ new-proposal ]`
- **أرشفة تلقائية (Auto-Archiving):** أرشفة الملفات القديمة تلقائيًا بناءً على سياسات مع إمكانية الاسترجاع `[ new-proposal ]`
- **تحليلات وتقارير (Analytics/Reports):** تقارير استخدام تفصيلية لكل مؤسسة `[ new-proposal ]`
- **تكامل LMS:** تكامل مع منصات إدارة التعلم الشائعة (Moodle بشكل أساسي) عبر LTI أو API مخصّص `[ new-proposal ]`
- **واجهة برمجة تطبيقات عامة (Public API):** REST API موثّقة (OpenAPI/Swagger) تسمح للمطورين بالتكامل مع ابن الأزهر دوكس برمجيًا `[ new-proposal ]`
- **دعم Word وPowerPoint كملفات إدخال:** توسيع صيغ الإدخال المقبولة `[ new-proposal ]`
- **تحسين OCR بالذكاء الاصطناعي (AI OCR Enhancement):** استخدام نموذج لغوي (LLM) لتحسين نتائج OCR `[ new-proposal ]` `[ needs-verification ]`
- **تصنيف تلقائي للمستندات (Auto-Classification):** تصنيف المستندات تلقائيًا حسب الموضوع أو النوع `[ new-proposal ]` `[ needs-verification ]`
- **تطبيق سطح المكتب (Tauri):** إعادة بناء تجربة سطح المكتب مستوحاة من tahweel-tauri الأصلي كمرجع بحثي `[ repo-reference ]` باستخدام Tauri 2 `[ new-proposal ]`

### ما هو خارج النطاق

- تطبيقات الهاتف الأصلية (Enterprise/Future)
- SSO/SAML (Enterprise/Future)
- White-label (Enterprise/Future)

### التبعيات

- تصميم نظام المؤسسات (البيانات، الأذونات، الفوترة المحتملة)
- تكامل LTI مع Moodle (يحتاج بيئة اختبار Moodle)
- نموذج AI للتحسين والتصنيف (قد يكون نموذج محلي أو API خارجي)
- بيئة تطوير Tauri 2 (Rust toolchain)
- مواصفات API موثّقة ومجموعة اختبارات

### المخاطر

| الخطر | الاحتمالية | التخفيف |
|---|---|---|
| نظام المؤسسات يزيد التعقيد بشكل كبير | عالية | تصميم بسيط في البداية: مؤسسة ← أقسام ← أعضاء فقط |
| تكامل LMS يتطلب دعم إصدارات متعددة من Moodle | متوسطة | البدء بـ Moodle 4.x فقط |
| AI OCR Enhancement يُنتج نتائج غير موثوقة | عالية | عرضه كخيار اختياري مع مراجعة المستخدم |
| تطبيق Tauri يحتاج صيانة مستقلة عن الويب | متوسطة | مشاركة أقصى قدر من كود الويب؛ استخدام Tauri كغلاف فقط |

### معايير النجاح

1. 3 مؤسسات تعليمية تستخدم النظام بشكل فعال

### الهدف

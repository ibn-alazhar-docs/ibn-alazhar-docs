## Epic 2: المصادقة (Auth)

### الهدف

بناء نظام مصادقة كامل مع تسجيل، دخول، أدوار (طالب/معلم/مشرف)، وجلسات آمنة عبر NextAuth.js v5 باستخدام JWT strategy فقط (مدة 24 ساعة). لا يوجد access/refresh token منفصل — JWT هو آلية الجلسة الوحيدة. يُمكّن هذا Epic التحكم في الوصول لجميع وظائف المنصة.

### قصص المستخدم

| # | القصة |
|---|-------|
| US-2.1 | بصفتي مستخدمًا جديدًا، أريد إنشاء حساب بالبريد الإلكتروني وكلمة المرور حتى أستخدم المنصة |
| US-2.2 | بصفتي مستخدمًا، أريد تسجيل الدخول ببياناتي حتى أصل لملفاتي من أي جهاز |
| US-2.3 | بصفتي مستخدمًا، أريد تسجيل الخروج من جميع الأجهزة حتى أحمي حسابي |
| US-2.4 | بصفتي مشرفًا، أريد أن يكون لديّ صلاحيات أعلى من المستخدمين العاديين حتى أدير المنصة |
| US-2.5 | بصفتي مستخدمًا، أريد أن تظل جلستي فعّالة أثناء تصفحي حتى لا أضطر لتسجيل الدخول مرارًا |
| US-2.6 | بصفتي مشرفًا، أريد إدارة أدوار المستخدمين حتى أمنح الصلاحيات المناسبة |

### المهام

| # | المهمة | التقدير (ساعات) |
|---|-------|----------------|
| T-2.1 | إعداد NextAuth.js v5 مع PrismaAdapter — JWT strategy فقط (24h) | 3 |
| T-2.2 | تنفيذ Credentials provider مع bcrypt | 2 |
| T-2.3 | إنشاء صفحة تسجيل (/register) مع Zod validation | 3 |
| T-2.4 | إنشاء صفحة دخول (/login) | 2 |
| T-2.5 | تنفيذ JWT strategy مع role claims — لا access/refresh token منفصل | 2 |
| T-2.6 | تنفيذ JWT callbacks (session update، role refresh) | 2 |

| T-2.10 | إنشاء Prisma seed script (admin + test users) | 1 |
| T-2.11 | إضافة سجل النشاط (ActivityLog) للأحداث الأمنية | 2 |
| T-2.12 | تنفيذ حماية CSRF (NextAuth.js v5 built-in) | 0.5 |
| T-2.13 | اختبارات وحدة للمصادقة (Vitest) | 3 |

**الإجمالي**: ~25.5 ساعة

### معايير القبول

- **Given** صفحة التسجيل، **When** يُدخل المستخدم بيانات صالحة، **Then** يُنشأ حساب بدور "طالب" ويُوجَّه لصفحة الترحيب
- **Given** حساب موجود، **When** يُدخل المستخدم بيانات صحيحة، **Then** يُوجَّه لصفحته الرئيسية مع جلسة JWT فعّالة (24 ساعة)
- **Given** مستخدم بدور "طالب"، **When** يحاول الوصول لصفحة الإدارة، **Then** يرى خطأ 403 Forbidden
- **Given** جلسة فعّالة، **When** يمر 4 ساعات، **Then** تُحدَّث الجلسة تلقائيًا دون إعادة تسجيل دخول (JWT updateAge)

### الاعتماديات

- **Epic 1** (Foundation): يجب إكمال Prisma + Docker + هيكل المشروع أولًا

### التصنيف

| الأولوية | التعقيد | Phase |
|---------|--------|--------|
| **P0** | **L** | Phase 1 (هيكل أساسي) / Phase 2 (اكتمال) |

### المخاطر

1. **تغييرات NextAuth.js v5 API** — المكتبة لا تزال في beta، قد تتغير الـ APIs
2. **مهاجمة حسابات ضعيفة** — حل: bcrypt + rate limiting على محاولات الدخول
3. **JWT role staleness** — حل: تحديث الـ role عند كل session update (updateAge trigger)

تطبيق طبقات أمان متعددة تشمل: CSP headers، rate limiting، تحقق مدخلات شامل، حماية من XSS/CSRF/SQLi، تشفير، وتدقيق أمني. يُنفّذ هذا Epic بشكل تدريجي عبر جميع Phases مع تركيز في Phase 5-6.

### قصص المستخدم

| # | القصة |
|---|-------|
| US-11.1 | بصفتي مستخدمًا، أريد أن تكون بياناتي محمية من الوصول غير المصرح به |
| US-11.2 | بصفتي مشرفًا، أريد rate limiting لحماية المنصة من الهجمات |
| US-11.3 | بصفتي مستخدمًا، أريد أن يُحظر النص البرمجي الضار (XSS) |
| US-11.4 | بصفتي مطوّرًا، أريد تحقق مدخلات شامل لجميع API endpoints |

### المهام

| # | المهمة | التقدير (ساعات) |
|---|-------|----------------|
| T-11.1 | تنفيذ CSP headers عبر Next.js middleware + Caddy | 2 |
| T-11.2 | تنفيذ Rate Limiting middleware (Redis-based) | 3 |
| T-11.3 | تنفيذ Zod validation لجميع API endpoints | 4 |
| T-11.4 | تنفيذ sanitization لمحتوى OCR | 2 |
| T-11.5 | تنفيذ IDOR protection لكل API route | 3 |
| T-11.6 | تنفيذ account lockout (5 محاولات فاشلة → قفل 15 دقيقة) | 2 |
| T-11.7 | مراجعة أمنية ذاتية (OWASP Top 10) | 4 |
| T-11.8 | اختبارات أمنية أساسية (Vitest + يدوي) | 3 |

**الإجمالي**: ~23 ساعة

### معايير القبول

- **Given** أي API endpoint، **When** يُرسل طلب بدون session صالح، **Then** يُرفض بـ 401
- **Given** مستخدم عادي، **When** يحاول الوصول لملف مستخدم آخر، **Then** يُرفض بـ 403
- **Given** 5 محاولات دخول فاشلة، **When** يُحاول مرة سادسة، **Then** يُقفل الحساب لـ 15 دقيقة
- **Given** XSS payload في مدخلات، **When** يُعرض النص، **Then** لا يُنفّذ السكربت

### الاعتماديات

- **Epic 2** (Auth): أساس المصادقة
- **Epic 1** (Foundation): Caddy reverse proxy

### التصنيف

| الأولوية | التعقيد | Phase |
|---------|--------|--------|
| **P0** | **L** | تدريجي (Phase 1-6)، تركيز Phase 5-6 |

### المخاطر

1. **CSP قد يكسر مكوّنات طرف ثالث** — حل: report-only أولاً ثم تفعيل
2. **Rate limiting قد يؤثر على المستخدمين الشرعيين** — حل: حدود معقولة + whitelist

تحويل المنصة إلى PWA مع Service Worker و manifest.json و Install Prompt و Offline Shell. يستخدم @serwist/next كاقتراح قيد الدراسة (لم يُحسم القرار النهائي بعد). يضمن هذا Epic تجربة تطبيق أصلي مع إمكانية التثبيت والعمل الأساسي دون اتصال.

### قصص المستخدم

| # | القصة |
|---|-------|

| US-12.2 | بصفتي مستخدمًا، أريد أن يعمل التطبيق بشكل أساسي دون اتصال حتى أتصفح ملفاتي عند ضعف الإنترنت |
| US-12.3 | بصفتي مستخدمًا، أريد رسالة تثبيت مخصصة بالعربية حتى أفهم كيف أثبّت التطبيق |
| US-12.4 | بصفتي مستخدمًا، أريد أن تظهر أيقونة التطبيق بشكل صحيح على جهازي |
| US-12.5 | بصفتي مستخدمًا، أريد واجهة "غير متصل" واضحة عند فقدان الاتصال |

### المهام

| # | المهمة | التقدير (ساعات) |

| T-12.5 | إنشاء صفحة Offline | 2 |
| T-12.6 | تنفيذ Install Prompt مخصص | 2 |

| T-12.9 | تنفيذ تحديث Service Worker مع إشعار المستخدم | 2 |

### معايير القبول

- **Given** المستخدم زار الموقع 3 مرات، **When** يفتح الموقع مرة رابعة، **Then** يرى Install Prompt مخصص بالعربية
- **Given** التطبيق مُثبَّت، **When** يُفتح من الشاشة الرئيسية، **Then** يعمل بوضع standalone (لا شريط عنوان المتصفح)
- **Given** فقدان الاتصال، **When** يتصفح المستخدم، **Then** يرى App Shell + صفحة Offline عند عدم توفر محتوى مخزّن
- **Given** Service Worker جديد، **When** يتوفر تحديث، **Then** يرى المستخدم إشعار "تحديث متاح"

### الاعتماديات

- **Epic 1** (Foundation): هيكل Next.js
- **Epic 3** (Design System): صفحة Offline تحتاج Design System

### التصنيف

| الأولوية | التعقيد | Phase |
|---------|--------|--------|
| **P1** | **M** | Phase 5 |

### المخاطر

1. **سلوك PWA يختلف بين المتصفحات** — حل: اختبار مكثّف على Chrome + Safari + Firefox
2. **Service Worker caching problems** — حل: versioned cache names + skipWaiting
3. **@serwist/next قد لا يكون مستقرًا كفاية** — حل: تقييم شامل + fallback لـ SW مكتوب يدويًا
4. **Install Prompt API غير متاح على جميع المتصفحات** — حل: fallback لتعليمات تثبيت يدوي
